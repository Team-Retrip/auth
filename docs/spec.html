Retrip Auth 도메인 — API & 화면 상태 명세서

auth 도메인의 전체 API 엔드포인트와 각 화면의 상태 정의를 담은 명세서입니다.
현재 구현 완료된 항목과 추가 개발이 필요한 항목을 함께 포함합니다.

✅ 구현 완료 — 현재 코드에 존재 🔧 개발 필요 — 이번 스프린트 추가 구현 🎨 프론트 처리 — 백엔드 변경 없음 📋 다음 배포 — 차기 스프린트
ℹ️
Base URL: http://localhost:8080 (로컬) / https://api.retrip.io (운영)
인증 방식: JWT Bearer Token — Authorization: Bearer {accessToken}
Content-Type: application/json

2B
소셜 로그인 3단계 처리 로직 ✅ v2.1 구현완료

✅
feat/social-login-multi-provider-nickname 브랜치 반영 (2026-04-02)
소셜 로그인 시 OAuth2LoginSuccessHandler가 아래 3단계로 처리합니다.
member.provider_id 컬럼이 제거되고 member_social_provider 테이블로 분리됩니다.
처리 흐름
단계조건처리
1단계
기존 소셜 계정		 member_social_provider 테이블에 동일 providerId 존재 로그인 처리 + updateLastLoginProvider(provider) 호출
2단계
이메일 자동 연동		 동일 이메일 기존 계정 존재 (로컬 또는 다른 소셜) 기존 계정에 새 소셜 providerId 추가 (MemberSocialProvider 신규 저장)
3단계
신규 가입		 해당 이메일 계정 없음 신규 Member 생성 + MemberSocialProvider 저장
DB 스키마 변경 (운영 배포 시 수동 DDL 필요)
-- 신규 테이블 (Hibernate ddl-auto: update 로 자동 생성됨)
CREATE TABLE member_social_provider (
  id          BIGINT AUTO_INCREMENT PRIMARY KEY,
  member_id   VARCHAR(36) NOT NULL,
  provider    VARCHAR(20) NOT NULL,  -- "kakao" | "naver" | "google"
  provider_id VARCHAR(255) NOT NULL
);

-- 기존 컬럼 제거 (수동 실행 필요 — ddl-auto: update는 컬럼 DROP 안 함)
ALTER TABLE member DROP COLUMN provider_id;

-- 신규 컬럼 (Hibernate ddl-auto: update 로 자동 추가됨)
ALTER TABLE member ADD COLUMN nickname VARCHAR(15);
주요 설계 원칙
⚠️
hasPassword() 패턴 — 반드시 준수
소셜 전용 계정 여부를 판단할 때 provider == "local" 비교를 절대 사용하지 않습니다.
항상 member.hasPassword() 메서드를 사용하세요 — 이 메서드는 password != null을 검사합니다.
GET /users/me의 hasPassword 필드를 프론트에서 소셜 유저 UI 분기에 사용하세요.

소셜 로그인 정책

항목정책
